# World Contract Audit

This audit covers the map/world mismatch found after moving multiplayer from the Node.js backend to the Go backend.

## Root Cause

The old Node.js backend and the PixiJS client both used `packages/shared/src/generation.ts` and `packages/shared/src/physics.ts`.

The Go backend introduced its own world generator in `apps/server-go/internal/game/world.go`. The client still bootstrapped chunks locally with the TypeScript generator and only requested a chunk from the server when `loadedChunks` did not already contain that chunk.

That created a mixed world:

- the client rendered and predicted against TypeScript-generated platforms, relics, jump pads, and local collision tiles;
- the Go server simulated Go-generated platforms, relics, jump pads, enemies, and collisions;
- chunks that were locally generated first could remain unrequested forever.

This explains the cluster of symptoms:

- enemies appeared to walk in the air because their server platform did not match the rendered client platform;
- players could phase through apparent platforms because the client-rendered platform was not the server platform;
- collectibles were not picked up because the server trigger was in a different place;
- invisible jump pads existed because the Go trigger was present where the client had no rendered pad;
- some objects existed only on one side of the connection.

## Source Of Truth

| Layer | Current source before fix | After fix |
| --- | --- | --- |
| Old Node backend | Shared TypeScript `generateVerticalChunk` | Reference only |
| Go backend | Go `GenerateChunk` | Authoritative multiplayer source |
| PixiJS client rendering | Local TS generation plus server chunks | Server chunks when connected, TS generation only as fallback/offline bootstrap |
| PixiJS client prediction | `loadedChunks` tile map | Same `loadedChunks`, now replaced by server chunks in active multiplayer window |
| Gameplay object state | Server owns pickups, enemies, jump pads, players | unchanged |

## Coordinate Rules

| Rule | Contract |
| --- | --- |
| Units | World units are pixels |
| Tile size | `TILE_SIZE` / `TileSize` = 16 px |
| Chunk size | 24 x 18 tiles |
| Y axis | Positive Y points downward |
| Chunk stacking | `worldTileY = -chunkY * CHUNK_HEIGHT_TILES` |
| Tile origin | `(tileX * TILE_SIZE, (chunk.worldTileY + tileY) * TILE_SIZE)` |
| Player position | Top-left of collision box |
| Enemy position | Top-left of collision box |
| Relic trigger | Center at tile center, 20 px pickup radius |
| Jump pad trigger | Center at tile center, width `1.7 * TILE_SIZE`, height `1.8 * TILE_SIZE` |
| Platform collision | One-way surface at platform tile row top edge |
| Solid floor | Chunk 0 bottom row only |

## Object Compatibility Matrix

| Object type | Node.js semantics | PixiJS client usage | Go backend semantics | Status / fix |
| --- | --- | --- | --- | --- |
| Platforms | Generated from shared TS chunks; one-way spans in `platforms` and `tiles` | Renders from `chunk.platforms`; predicts via `tileMap` | Generated in Go with compatible JSON shape, different layout | Server-authoritative chunks now replace local chunks |
| One-way platforms | Collide only when falling from above; drop input bypasses | Local prediction uses shared physics | Go collision uses top-edge landing and drop bypass | Compatible semantics, requires same chunk data |
| Floor | Solid bottom row only in chunk 0 | Rendered from `tiles` | Same | Compatible |
| Enemies | Hydrated from shared chunk spawns, server state sent in snapshots | Rendered from snapshots | Hydrated from Go chunk spawns | Compatible only after client displays server chunks |
| Collectibles/relics | Server pickup radius at rendered relic location | Rendered from chunk relic list, removed by events | Same trigger radius and event shape | Fixed by server-authoritative chunks |
| Jump pads | Trigger near rendered pad center | Rendered from chunk jump pad list; local FX uses chunk pad list | Same trigger shape, Go multiplier may differ by generator | Fixed by server-authoritative chunks |
| Wind zones | Generated by TS generator | Client renders/applies local wind zones | Go currently sends empty `windZones` | Missing gameplay parity if wind zones are required |
| Hazards | TS generator may place hazard tiles | Client renders hazard tiles | Go generator currently does not place hazard tiles | Missing gameplay parity if hazards are required |
| Portals/exits | Derived from chunk exit | Client renders portal/exit effects | Go exposes `exit` | Compatible for chunk data |
| Decorative props | Client procedural decoration | Client-only visuals | Not simulated | Allowed client-only |

## Collision Semantics

| Collision / trigger | Contract |
| --- | --- |
| Player vs platform | Server is authoritative. Client prediction must use the same active chunk data received from server. |
| Drop-through | Input `drop` skips one-way platform landing for that tick. |
| Relic pickup | Server checks player center against relic center within 20 px on both axes. |
| Jump pad | Server checks player bottom near pad center while falling/not rising. |
| Enemy ground | Enemy `platformY`, patrol min/max, and spawn position derive from server chunk platform data. |
| Client visuals | Must render the same chunk data the server generated for collision-relevant objects. |

## Fixes Implemented

- Added `authoritativeChunks` and `pendingAuthoritativeChunks` in `apps/client/src/main.ts`.
- Changed chunk requests so the client requests server chunks even when a local fallback chunk already exists.
- Marked received `chunk` messages as authoritative and replaced local visuals/state.
- Cleared authoritative/pending markers when chunks are pruned or the world resets.
- Added an F1 world debug overlay showing:
  - server vs local chunk source,
  - platform collision boxes,
  - floor collision,
  - relic trigger zones and IDs,
  - jump pad trigger boxes and IDs,
  - enemy hitboxes, ground probes, and patrol lines,
  - player authoritative and rendered boxes.
- Added `GET /debug/world?room=demo&chunk=0` and range queries (`minChunk`, `maxChunk`, `radius`) to dump server world data.
- Added regression tests for the debug dump and endpoint.

## Debug Endpoint

Examples:

```sh
curl "http://localhost:8787/debug/world?room=demo&chunk=0"
curl "http://localhost:8787/debug/world?room=demo&chunk=4&radius=1"
curl "http://localhost:8787/debug/world?room=demo&minChunk=0&maxChunk=5"
```

The dump includes tile size, coordinate rules, platforms, solid tiles, relic trigger zones, jump pad trigger boxes, enemy patrol surfaces, and object IDs.

## Remaining Recommendation

The server-authoritative chunk flow fixes the multiplayer mismatch without rewriting gameplay rendering. The next durable improvement is to move the map schema/generator into one shared artifact or make the Go server the only generator and always send chunk data before gameplay begins. Avoid maintaining independent TypeScript and Go procedural generators unless a reconciliation test proves byte-for-byte object equivalence.
